## Story

**As a** developer,  
**I want** the desktop MVP components fully designed and specified,  
**so that** Epic 1 can be implemented using the repo stack without UI decisions being improvised

---

## Acceptance Criteria

- [x] Component inventory covers only current MVP and Epic 1 workflows
- [x] Each component defines desktop interaction states where applicable
- [x] Components include accessibility notes relevant to implementation
- [x] Component guidance maps each item to a shadcn/ui primitive or project wrapper approach
- [x] No collaboration-only components are required for MVP
- [x] Repo handoff documentation and naming match code terminology

---

## Tasks / Subtasks

- [x] Design core form controls: input, textarea, select, checkbox, radio, switch, tag input
- [x] Design action components: primary button, secondary button, ghost button, icon button, loading button
- [x] Design shell/navigation components: sidebar, top bar, tabs, breadcrumbs
- [x] Design data display components: card, list row, table surface, badge, chip, empty state, skeleton
- [x] Design feedback components: alert, toast, modal, progress, sync status
- [x] Design Epic 1 specializations: file browser card, metadata panel, version timeline, rollback dialog, export state
- [x] Document state behavior, variant rules, and accessibility notes for each family
- [x] Produce a stack mapping note for each family: shadcn primitive vs project wrapper

### Component Scope Rules

- Do not include avatar, team presence, or collaboration-only widgets
- Do not treat mobile/tablet variants as MVP acceptance criteria
- Use shadcn/ui primitives as the baseline whenever possible

### Example Mapping

- Button -> shadcn button variants
- Input / textarea / select -> shadcn form primitives
- Dialog -> shadcn dialog
- Tabs -> shadcn tabs
- Toast -> shadcn toast
- Table -> project wrapper around a desktop table pattern
