## Story

**As a** developer,  
**I want** to correct design-system drift in the current MVP UI and extract repeated patterns into reusable components,  
**so that** the application becomes visually consistent, easier to maintain, and aligned with the approved Epic 0 guides

---

## Acceptance Criteria

- [x] All currently implemented MVP surfaces use approved semantic tokens or shared semantic classes when an approved equivalent already exists
- [x] `src/pages/index.tsx` is remediated to the Epic 0 design-system baseline
- [x] Repeated UI structures are extracted into reusable atoms, molecules, or organisms only where repetition is real and implementation-facing
- [x] Existing atomic exports reflect the reusable components introduced by the remediation
- [x] The remediation reduces duplicated visual patterns across pages and components without inventing new product scope
- [x] The remediation does not reintroduce mobile-first, premium styling, collaboration UI, or a parallel design system

---

## Tasks / Subtasks

- [x] Use the Story 0.7 audit as the source of truth for remediation order
- [x] Replace ad hoc token usage with approved semantic tokens or shared semantic classes
- [x] Remediate the home page to the same semantic design-system baseline already used in the Epic 1 seed surfaces
- [x] Extract repeated status, dialog, layout, and action-group patterns only where duplication is confirmed
- [x] Update `src/components/{atoms,molecules,organisms}/index.ts` to reflect the reusable components introduced
- [x] Align implementation naming to the approved design-system and handoff guides
- [x] Update affected tests or add coverage where extracted components materially change rendering structure

[Source: docs/guides/accessibility-compliance.md]
- Accessibility semantics must be preserved or improved during remediation

### Implementation Boundaries

- No backend API contract changes are expected
- Reusable extraction is allowed only when a pattern is present in at least two relevant MVP surfaces
- Prefer wrappers over Tailwind + tokens and semantic classes instead of introducing a new abstraction layer
- Do not create future-facing components that are not justified by the current UI and approved product artifacts

### Expected Focus Areas

- Home page semantic alignment
- Reusable alert and status-surface patterns
- Reusable dialog shell structure
- Reusable shell/info-card or action-group patterns where duplication is meaningful
- Cleanup of direct gray/white utilities when semantic alternatives already exist

### Execution Summary

- Used Story 0.7 as the remediation order and resolved the only fully non-compliant MVP surface by rebuilding `src/pages/index.tsx` on the approved semantic baseline
- Extracted reusable `StatusNotice` and `DialogShell` components only where repetition was already present in the implemented UI
- Replaced repeated inline status and dialog shells in `src/pages/creative-library.tsx` and `src/components/CreativeLibrary/MetadataForm.tsx`
- Updated `OfflineIndicator` to use the approved `lucide-react` icon policy instead of a literal glyph
- Added focused tests for the new reusable components and reran all required quality gates successfully
