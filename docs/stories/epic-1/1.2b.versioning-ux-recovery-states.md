## Story

**As a** creative operator,  
**I want** versioning state to feel transparent and recoverable,  
**so that** queued, retrying, failed, and refresh-required situations remain understandable before I use history and rollback

---

## Acceptance Criteria

1. The UI consumes the versioning lifecycle emitted by `1.2` instead of inventing a second queue model.
2. The interface differentiates local-save success from final version-history success when versioning completes asynchronously.
3. User-facing state covers queued, committing, retrying, failed, and completed versioning work.
4. Refresh-required and stale-state behavior follows `docs/concurrent-editing-strategy.md` and `docs/error-handling-spec.md`.
5. Overflow, retry exhaustion, and terminal failure use the approved error semantics instead of ad hoc UI messaging.
6. This story explicitly does not implement Epic 7 offline queue persistence or sync behavior.
7. The result provides a stable async-state foundation for `1.3` and `1.4`.
8. Lint, typecheck, tests, and build pass after the implementation.
