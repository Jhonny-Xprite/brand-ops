## Story

**As a** creative operator,  
**I want** a browsable creative library UI with upload, navigation, and search/filter controls,  
**so that** I can manage files from one central screen before version-history features are layered on top

---

## Acceptance Criteria

1. The creative library screen renders a file-browser layout with sidebar, top controls, main file area, and details panel shell.
2. Users can upload supported files from the OS into the approved local storage root `E:\BRAND-OPS-STORAGE\` and new `CreativeFile` records are created with persisted paths under that contract.
3. Grid/list view toggle works and persists in local storage.
4. Search and filter controls exist for filename, type, and status, even if advanced filter polish comes later.
5. Thumbnail handling supports images automatically, falls back safely for unsupported preview types, and does not break the file list.
6. Multi-select is supported for at least the baseline file-card/list interactions described in the Epic 1 PRD.
7. Keyboard navigation and focus behavior cover the primary browse interactions.
8. User-visible error handling for upload and filesystem failures follows `docs/error-handling-spec.md`.
9. The story does not implement git-backed version history yet; it prepares the UI and data flow for later stories.
10. Lint, typecheck, tests, and build pass after the implementation.
