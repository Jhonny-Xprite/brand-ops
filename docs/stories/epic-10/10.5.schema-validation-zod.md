## Risks

**Technical Risks:**

1. **API Breaking Changes** (MEDIUM impact, HIGH probability)
   - Existing API clients may receive different error formats
   - Mitigation: Implement dual validation (legacy + Zod) for 1 sprint, then migrate
   - Effort: +0.5 days for backward compatibility layer

2. **Migration Complexity** (MEDIUM impact, MEDIUM probability)
   - 15+ API endpoints need refactoring
   - Multiple form components need integration
   - Mitigation: Use phased rollout (Phase A, B, C) if needed
   - Current approach: All-at-once (simpler, higher risk but faster)

3. **Schema Scope Creep** (LOW impact, MEDIUM probability)
   - Adding validations for every field may expand scope significantly
   - Mitigation: Define MVP schemas first, extend in follow-up stories
   - Decision: Strict AC compliance only

4. **Type Inference Complexity** (LOW impact, LOW probability)
   - TypeScript type inference from Zod may cause compilation issues
   - Mitigation: Use `z.infer<typeof Schema>` pattern consistently
   - Testing: Full typecheck pass required before merge

## Example Schema Structure

```typescript
// src/lib/schemas/files.ts
import { z } from 'zod'

export const UpdateFileMetadataSchema = z.object({
  type: z.string().min(1, 'File type is required'),
  status: z.string().min(1, 'Status is required'),
  tags: z.array(z.string()).max(20, 'Maximum 20 tags'),
  notes: z.string().max(500, 'Notes must be under 500 characters').optional(),
})

export type UpdateFileMetadata = z.infer<typeof UpdateFileMetadataSchema>
```

**API Usage:**
```typescript
// src/pages/api/files/[id].ts
const parsed = UpdateFileMetadataSchema.safeParse(req.body)
if (!parsed.success) {
  return res.status(400).json({ errors: parsed.error.errors })
}
const data = parsed.data
// ... proceed with validated data
```

**Form Usage:**
```typescript
import { zodResolver } from '@hookform/resolvers/zod'

const form = useForm({
  resolver: zodResolver(UpdateFileMetadataSchema),
  // ...
})
```

### Architecture Decisions

1. **Schema Organization**: Three-tier structure (common → domain → exports)
   - Rationale: Clear separation of concerns, reusable base schemas
   
2. **Validation Error Format**: Zod's flatten() with field-level details
   - Rationale: Matches REST API best practices for validation errors
   
3. **Optional Fields in Update Schemas**: All fields optional in Update schemas
   - Rationale: Supports partial updates without requiring unused fields
   
4. **Type Inference**: Full use of z.infer<typeof Schema>
   - Rationale: Single source of truth for types + runtime validation
