## Story

**As a** developer,  
**I want** a Service Worker registered with basic offline detection,  
**so that** the app can detect offline/online state and handle it gracefully

---

## Acceptance Criteria

### AC1: Service Worker Registration

- **Given** the app initializes in a browser with Service Worker support
- **When** the root layout component mounts
- **Then** `navigator.serviceWorker.register('/sw.js')` is called and logs success to console

### AC2: Service Worker File & Lifecycle

- **Given** the Service Worker is registered
- **When** the browser installs/activates the worker
- **Then** `install` and `activate` handlers execute with `skipWaiting()` and `clients.claim()`

### AC3: Online Status Detection Hook

- **Given** a component uses `useOnlineStatus` hook
- **When** `navigator.onLine` is accessible
- **Then** hook returns current online status and updates on `online`/`offline` events

### AC4: Offline Indicator Component

- **Given** the app is offline (network disabled in DevTools)
- **When** the OfflineIndicator component renders
- **Then** displays "🔴 OFFLINE" in red color in the page header

### AC5: Online Indicator Component

- **Given** the app is online (network re-enabled)
- **When** the OfflineIndicator component renders
- **Then** displays "🟢 ONLINE" in green color in the page header

### AC6: Event Listener Cleanup

- **Given** a component using `useOnlineStatus` unmounts
- **When** the effect cleanup runs
- **Then** `online` and `offline` event listeners are removed (no memory leaks)

### AC7: Build Verification

- **Given** all Service Worker and hook code is implemented
- **When** `npm run typecheck` and `npm run build` execute
- **Then** both succeed with zero errors and warnings

### AC8: SSR-Safe Implementation

- **Given** the hook is used in a Next.js SSR context
- **When** code checks `typeof navigator` before access
- **Then** no runtime errors occur during server-side rendering

### AC9: Service Worker Persistence

- **Given** Service Worker is registered and active
- **When** page is refreshed or reloaded
- **Then** Service Worker remains registered (visible in DevTools → Application → Service Workers)

### AC10: Error Handling

- **Given** Service Worker registration or browser API fails gracefully
- **When** error occurs (older browser without Service Worker API)
- **Then** app continues functioning without offline detection feature

---

## Risks

- **Browser Compatibility Risk:** Service Worker API not available in older browsers (IE 11). Mitigation: Graceful fallback with try-catch blocks
- **Service Worker Lifecycle Complexity:** SW updates can cause unexpected behavior if not properly managed. Mitigation: Proper use of `skipWaiting()` and `clients.claim()`
- **Event Listener Memory Leaks:** Forgotten event listener cleanup can accumulate listeners on re-renders. Mitigation: Implement proper cleanup in useEffect hook
- **HTTPS Requirement:** Service Workers only work on HTTPS (or localhost). Mitigation: Document for production deployment
- **Offline Detection Edge Cases:** Network state may not reflect actual connectivity (e.g., DNS failures). Mitigation: Acknowledge as known limitation

### Service Worker Basics

- Service Workers run in background on a separate thread
- Cannot access DOM
- Can intercept network requests
- Can cache responses
- Can handle offline scenarios
- Must be served from HTTPS (localhost OK for dev)

### Browser Compatibility

- `navigator.serviceWorker` - modern browsers only
- Graceful fallback: if not available, app still works
- `navigator.onLine` - supported in all modern browsers
